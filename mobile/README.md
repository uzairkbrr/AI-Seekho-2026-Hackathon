# CIRO Mobile — Field Operations App

The Flutter client for the **Crisis Intelligence & Response Orchestrator (CIRO)**. It is the primary interface field responders, dispatchers, and coordinators use to monitor live crises, inspect agent reasoning, allocate resources, and run response simulations.

See the [project root README](../README.md) for full system context.

## Stack

- **Flutter** `^3.11.5` (Dart 3, Material 3, dark theme)
- **Riverpod 3** for state and async data
- **`http`** for the JSON API client
- **`fl_chart`**, **`shimmer`**, **`google_fonts`**, **`intl`** for UI polish

Pinned in [pubspec.yaml](pubspec.yaml).

## Requirements

- Flutter SDK `^3.11.5` (`flutter --version`)
- Android SDK + emulator, an iOS simulator, or a physical device
- The CIRO FastAPI backend running on the host on port `8000` — see [`../backend/README.md`](../backend/README.md)

## Running locally

1. Start the backend (in a separate terminal):
   ```bash
   cd ../backend
   uvicorn app.main:app --host 0.0.0.0 --port 8000
   ```
2. Install Flutter packages and launch the app:
   ```bash
   flutter pub get
   flutter run
   ```

The default base URL is `http://10.0.2.2:8000/api`, which is the Android emulator's loopback to the host. Adjust per target — see [Backend URL](#backend-url) below.

### Backend URL

The base URL is a single constant in [lib/core/api.dart](lib/core/api.dart):

| Target           | `_base`                            |
| ---------------- | ---------------------------------- |
| Android emulator | `http://10.0.2.2:8000/api` (default) |
| iOS simulator    | `http://localhost:8000/api`        |
| Physical device  | `http://<host-LAN-IP>:8000/api`    |
| Web (`-d chrome`) | `http://localhost:8000/api`       |

If the backend can't be reached, the dashboard renders shimmer placeholders and the Pipeline screen logs `Exception: GET /events failed`. Fix the URL or confirm the backend port and rerun.

## End-to-end walkthrough

The fastest way to see the system work end-to-end on a freshly started backend:

1. **Open Pipeline** (rightmost tab) and tap **Run Full Pipeline**. The six stages — Ingest → Fuse → Allocate → Simulate → Notify → Verify — execute sequentially, each surfacing the agent's reasoning trace in the execution log. Stages can also be run individually by tapping them.
2. **Switch to Home.** The shimmer placeholders resolve into live incidents pulled from `/events`. The KPI strip shows total events, critical count, and last-update time.
3. **Open Events → tap an incident.** The detail page renders:
   - **Impact Metrics** — affected population, radius, expected duration, fusion confidence
   - **Before / After Impact** — projected casualties, alerted population, hazard containment, and recovery time, both with and without the active response. Before the first Simulate, only the "no action" column is populated.
   - **Response Actions** — a custom-painted map of dispatches, closures, and alert zones from `/dispatches`, `/closures`, `/alert-zones`, plus per-unit ETAs.
   - **AI Assessment** — spread risk, peak impact time, contradiction level, uncertainty range, and the fusion agent's narrative.
   - **Signal Sources** — raw inputs (NASA / USGS / Weather / News) with credibility bars and suspicion flags from the verification agent.
4. **Tap Simulate** on the detail page. This `POST /simulate`s, refreshes the response state, and fills in the "with response" projections.
5. **Tap Advance** to `POST /simulate/tick` — dispatched units progress toward the scene, ETAs decrement, and on-scene status flips to green.
6. **Switch to Alerts** to read the notifications the orchestrator generated for each stakeholder group (public, responders, hospitals, traffic control).

## Architecture

```
lib/
├── main.dart                   # App shell + bottom nav
├── core/
│   ├── api.dart                # HTTP client (base URL + get/post helpers)
│   ├── models.dart             # Typed DTOs: CrisisEvent, Signal, DispatchTrack,
│   │                           # RoadClosure, AlertZone, Notification, Allocation
│   └── theme.dart              # Dark theme, brand colors, severity→color map
├── features/
│   ├── dashboard/page.dart     # Home: KPIs + severity heatmap
│   ├── events/
│   │   ├── page.dart           # Incident list
│   │   ├── detail.dart         # Per-incident view + Simulate / Advance controls
│   │   ├── provider.dart       # eventsProvider + eventResponseProvider(eventId)
│   │   └── response_map.dart   # CustomPainter map of dispatches/closures/zones
│   ├── resources/{page,provider}.dart
│   ├── notifications/{page,provider}.dart
│   └── pipeline/page.dart      # Stage runner + reasoning trace viewer
└── widgets/
    ├── event_card.dart
    ├── severity_badge.dart
    ├── shimmer.dart
    └── stat_card.dart
```

Riverpod providers own all network I/O. `FutureProvider` is used for list endpoints; `FutureProvider.family<T, int>` keys per-event state by `eventId` (see [features/events/provider.dart](lib/features/events/provider.dart)). Mutating actions (`Simulate`, `Advance`, individual pipeline stages) `POST` and then `ref.invalidate(...)` the relevant provider to refresh.

## Backend API surface used

Everything goes through `get(path)` / `post(path)` in [lib/core/api.dart](lib/core/api.dart):

| Method | Path                                  | Used by                                |
| ------ | ------------------------------------- | -------------------------------------- |
| GET    | `/events`                             | Home, Events list                      |
| GET    | `/closures?event_id={id}`             | Event detail (response map)            |
| GET    | `/dispatches?event_id={id}`           | Event detail (response map + list)     |
| GET    | `/alert-zones?event_id={id}`          | Event detail (response map + list)     |
| GET    | `/traces?stage={key}&since={iso}&limit=50` | Pipeline reasoning trace          |
| POST   | `/ingest` `/fuse` `/allocate` `/simulate` `/notify` `/verify` | Pipeline stages |
| POST   | `/simulate/tick`                      | Event detail "Advance" button          |

## Troubleshooting

- **Empty dashboard / `GET /events failed: 500`** — the backend started but no events exist yet. Open Pipeline and run Ingest + Fuse (or the full pipeline).
- **Connection refused on physical device** — the device can't reach `10.0.2.2`. Switch `_base` in [lib/core/api.dart](lib/core/api.dart) to your host's LAN IP and ensure your firewall allows port 8000.
- **Simulate button does nothing visible** — open Pipeline and check the execution log; if the `simulate` trace says `fallback`, the backend couldn't reach Gemini and used the deterministic fallback (still produces dispatches/zones/closures).
- **Stale data after running a stage** — Riverpod caches by provider; pull-to-refresh on the affected screen or hot-restart with `R`.
